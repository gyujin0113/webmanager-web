## 1. 전체 아키텍처

```
[고객] ──카톡(텍스트+사진)──> [규진]
                                │  요청문 붙여넣기 + 첨부 저장
                                ▼
                  [Claude Code + /apply-edit 스킬]  (규진 맥)
                                │  ① 고객 repo에서 브랜치 생성
                                │  ② content/ 파일 수정 (이미지는 R2 업로드 → URL 참조)
                                │  ③ next build 통과 검증
                                │  ④ PR 생성 (변경요약 + 원본 요청문)
                                ▼
                  [Cloudflare Pages 브랜치 프리뷰]  ← 자동 발급 별도 URL
                                │
                [사람게이트] 규진이 프리뷰 확인(폰/컴퓨터) → 필요시 고객에게 URL 전달
                                │  승인 = PR merge (폰 GitHub앱 또는 세션 gh pr merge)
                                ▼
                  [운영 배포 자동] · 롤백 = git revert (1수정 = 1PR = 1커밋)
```

- **고객당 repo 1개**, repo당 Pages 프로젝트 1개. 서버 0, DB 0.
- 신규 구축물 3가지: ① 콘텐츠 컨벤션(각 repo 적용) ② `/apply-edit` 스킬 ③ R2 업로드 스크립트.

## 2. 콘텐츠 컨벤션 (AI 안전 편집 구역)

```
customer-site/
├── content/                  ← ✅ AI 편집 허용 구역
│   ├── site.json             ← 상호·영업시간·전화·주소·SNS (구조화 값)
│   ├── sections/*.json|mdx   ← 섹션별 문구·가격표·메뉴
│   └── posts/*.mdx           ← 공지·블로그 (글 1개 = 파일 1개)
├── src/                      ← 🚫 AI 편집 금지 (코드·레이아웃·컴포넌트)
│   └── components/...           content/를 import만 함
└── public/                   ← 이미지 실물 없음. R2 URL만 content/에서 참조
```

- **수정범위 4티어(04 §4)와 폴더 경계가 1:1**: 티어①(무제한) = `content/` 안의 값 변경.
  티어③(별도견적) = `src/` 작업. 판정이 기계적으로 됨.
- 가드레일: `content/` 밖 변경이 필요한 요청이면 수정하지 않고 "별도 견적 대상" 보고.

## 2.5. 자동화 전제조건 — "모든 고객 콘텐츠는 content/로 분리" (필수 설계 규칙)

> **2026-06-15 첫 실전(grabis 영문 데이터 이미지 교체)에서 도출.** 이미지가 코드(`ProductPageLayout.tsx`)+`public/`에 하드코딩돼 있어 "이미지 교체"라는 흔한 티어① 요청이 자동화되지 못하고 수동 코드 작업으로 처리됐다. 이 사태의 재발을 막는 규칙.

**핵심 원칙: 고객이 바꿀 수 있는 모든 것(텍스트·값·이미지·영상·문서)은 그 *참조*가 반드시 `content/` 안의 데이터 파일에 있어야 한다.** 코드(`src/`·컴포넌트)에 하드코딩 금지.

자동 교체가 되려면 **두 조건을 동시에** 만족해야 한다:

1. **참조의 위치 = `content/`** (가장 중요)
   - 이미지/영상 경로, 표 데이터, 문구, 가격 등 — 전부 `content/*.json|mdx`에 둔다.
   - 컴포넌트(`src/`)는 `content/`를 import해서 렌더만 한다. 경로를 컴포넌트 안에 박지 않는다.
   - 이 조건이 깨지면(코드 하드코딩) AI가 가드레일 때문에 손도 못 댄다 → 티어③로 오분류.

2. **실물 바이너리의 위치 = R2** (git 비대화 방지, 04 §3.2)
   - 이미지·영상·PDF 실물은 R2(`assets.webmanager.co.kr`)에 올리고, `content/`엔 그 **공개 URL 문자열만** 둔다.
   - 작은 아이콘 등은 `public/`도 허용하나, 그 경우에도 **참조는 `content/`에** 있어야 교체 가능. 큰 이미지·영상은 R2 필수.

**적용 대상**
| 대상 | 조치 |
|---|---|
| 신규 고객 사이트 | 제작 시점부터 이 규칙 강제. 표준 스택 템플릿(§7-6)이 이 구조를 기본 골격으로 가진다. |
| 기존 grabis | 제품 데이터/SEM/어드밴티지 이미지 등 코드 하드코딩분을 **점진적으로 `content/`로 분리**(이미지 경로를 `content/sections/*.json`으로 이전 + 컴포넌트는 import). 분리 완료 전까지 해당 이미지 교체는 수동 처리. |
| goddmenz | 재구축 시 처음부터 이 규칙으로. |

**제작 체크리스트 (신규/리팩토링 공통)**
- [ ] 페이지에 보이는 모든 이미지/영상의 경로가 `content/`에서 오는가? (컴포넌트 grep으로 `.jpg|.png|.webp|.mp4` 하드코딩 0건 확인)
- [ ] 표·리스트·문구 등 데이터가 `content/`에 있는가?
- [ ] 큰 바이너리는 R2 URL로 참조하는가?
- [ ] `/apply-edit` 가드레일(`content/`만 변경)로 실제 교체가 되는지 1건 리허설했는가?

## 3. `/apply-edit` 스킬 워크플로

입력: 고객 식별자 + 카톡 요청문 (+ 첨부파일 경로)

1. **분류** — 4티어 판정. `content/`로 해결 불가 → 중단 + "티어③ 별도견적" 보고
2. **브랜치 생성** — `edit/YYYYMMDD-짧은설명`
3. **수정 적용** — `content/`만 변경. 이미지는 R2 업로드 스크립트 호출 → 반환 URL 참조
4. **검증** — `next build` 통과 + diff가 `content/` 안인지 **코드로** 검사 (AI 판단 아님)
5. **PR 생성** — 제목=요청 한 줄 요약 / 본문=변경 내역 + 원본 요청문 + 프리뷰 URL 안내
6. **보고** — PR 링크 + 프리뷰 URL 출력 → 규진 폰에서 확인·merge

## 4. R2 에셋 파이프라인

- 버킷 1개 `webmanager-assets`, 공개 접근은 커스텀 도메인(예: `assets.webmanager.co.kr`)
- 경로 컨벤션: `{고객slug}/{YYYY}/{파일명-해시}.webp` — 고객별 격리 + 캐시 영구화
- 업로드 스크립트 1개 (`wrangler r2 object put` 래핑): 파일 입력 → webp 변환·리사이즈(원본 보존) → 공개 URL 출력
- git에는 URL 문자열만. 바이너리 0 (04 §3.2).

## 5. 에러 처리 · 안전장치

| 상황 | 처리 |
|---|---|
| 요청이 `content/` 밖 요구 | 수정 없이 중단 → 티어③ 견적 안내 |
| `next build` 실패 | PR 미생성, 브랜치 폐기, 원인 보고 |
| diff에 `content/` 밖 파일 | PR 미생성, 중단 (기계적 검사) |
| 모호한 요청 ("더 예쁘게") | 추측 수정 금지. 되물을 질문 생성 (티어② 2회 규칙) |
| merge 후 문제 | `git revert` 1커밋 = 완전 롤백 → Pages 자동 재배포 |
| R2 업로드 실패 | 콘텐츠 수정 중단 (깨진 URL 커밋 방지) |
| 외부 위젯 장애 | 스킬 범위 밖 — '별도 복구' 분류 (04 §4.2) |

**원칙: 실패하면 아무것도 만들지 않는다.** 운영본은 merge 전까지 절대 변하지 않는다.

## 6. 테스트 · 검증

1. **리팩토링 무손상**: 콘텐츠 분리 전후 화면이 동일해야 함 (빌드 산출물 비교 + 프리뷰 육안)
2. **스킬 리허설** (가짜 시나리오 3종): ① 티어① "영업시간 변경" → PR 생성 ② 티어③ "예약 기능" → 중단해야 통과 ③ 모호 "더 예쁘게" → 되묻기 생성해야 통과
3. **엔드투엔드 본선**: 실제 카톡 요청 1건 → 스킬 → PR → 폰 승인 → 운영 반영 → 연습 revert 1회
4. **R2**: 업로드 → URL 200 → 사이트 렌더 확인

## 8. v2 진화 경로 (궁극 방향 — 합의됨)

> **고객이 포털에 직접 입력 → 헤드리스 에이전트가 자동 수정 → 포털에 프리뷰 URL 표시 →
> 고객이 직접 승인 → 자동 merge.** (01의 "B. 셀프서비스+가드레일")

v1 설계가 v2의 문을 열어두는 장치:
- **PR merge = 승인 프리미티브** — 고객 승인 버튼이 GitHub API merge를 호출하면 끝
- **`content/` 가드레일** — 사람 게이트 제거의 전제조건
- **스킬 로직 = 이식 가능한 워크플로** — v2에서 Agent SDK가 동일 절차를 헤드리스 실행

v1을 수동으로 시작하는 이유: 고객 월 1회 빈도에서 포털·자동화 선행 구축은 검증 전 비용. v1으로 절차 검증 후 "입력 폼 + 승인 버튼 + 헤드리스 실행"만 얹는다.
