## Problem Statement
코딩 모르는 소상공인(미용실 원장님, 진엽 형 회사 등)에게 맞춤 홈페이지를 만들어 납품한 뒤,
배포 후 끊임없이 들어오는 자잘한 수정 요청("영업시간 바꿔줘", "가격표 갱신", "사진 교체")을
사람이 손으로 쳐내느라 시간이 녹는다. 고객은 개발자에게 카톡하고 며칠 기다린다. 개발자(3&D)는
요청에 끌려다닌다. 둘 다 고통.

해결: 맞춤 홈페이지 제작 + 월 구독형 "무제한 수정" 유지보수. 수정의 90%를 AI가 쳐내서
무제한 수정을 수익성 있게 만든다.

## Demand Evidence
- 창업자(tm_lab)가 **이미 실제로 사이트를 2건 납품**: 본인이 다니는 미용실 원장님, 진엽 형 회사.
- 그 고객들이 **현재 진행형으로 수정 요청을 본인 폰으로 보내고 있음** = "사라지면 곤란" 상태(=실수요).
- 창업자 본인이 그 수정 노동의 고통을 직접 겪는 **첫 번째 사용자**.
- 빠진 것은 단 하나: **아직 돈을 받지 않음**. (= 다음 단계 검증 포인트)
- 약한 신호: 주변의 "있으면 좋겠다" 반응(관심 ≠ 수요로 분류).

## Status Quo (진짜 경쟁자)
- 끝단 사장님이 수정하고 싶을 때 현재 행동: **개발자(=창업자)에게 카톡하고 기다린다.**
- 더 넓은 status quo = Wix·스퀘어스페이스·아임웹·카페24·워드프레스 등 노코드/CMS 산업 전체.
  이미 "비전문가 셀프 편집"을 20년간 풀어왔고 지금 AI가 다 붙는 중(Wix AI, Framer AI, Durable).
- 따라서 "텍스트로 수정"만으로는 차별점 없음. **차별점은 묶음**:
  "우리가 *맞춤으로* 만들어주고(done-for-you, Wix는 셀프) + 그 위에 AI 셀프수정을 얹는다."

## Target User & Narrowest Wedge
- **돈 내는 고객**: 코딩 모르는 끝단 사장님(자영업·소상공인).
- **이중 수혜**: 사장님(빠른 수정 + 마음의 평화) + 3&D(요청 노동에서 해방, 구독 매출).
- **가장 좁은 쐐기**: 지금 너한테 카톡 보내는 미용실 원장님 + 진엽 형 회사. 새 시장 X.
  지인은 시장이 아니라 **세상에서 제일 싼 검증 채널**. 가격표를 들이밀어 진짜/가짜를 즉시 판별.

## Constraints
- 1인 창업(솔로). 빌드·운영 리소스 제한.
- 비전문가 고객은 코드 diff를 못 읽고 롤백도 못 함 → 신뢰가 절대 변수.
- 라이브 프로덕션 사이트를 자유 텍스트로 수정 → 안전이 제품의 생사.

## Premises
1. 진짜 제품은 "AI 수정"이 아니라 "구독형 무제한 유지보수". AI는 무제한 수정을 수익성 있게
   만드는 마진 엔진. (창업자 동의)
2. (수정됨) 지인은 *시장*이 아니라 *가장 싼 검증 채널*이다. 거기서 끝내는 게 아니라,
   거기에 **가격표를 들이밀어 검증**하고 소개로 확장한다. 검증 기준 = 돈 + 의존(사라지면 곤란).
   — 창업자가 "지인은 진짜 고객이 아니다"라고 반박 → 검증 기준을 '친구 vs 모르는 사람'이 아니라
   '돈 내는가 + 의존하는가'로 교정. 이 교정본에 합의.
3. 제품의 생사는 "AI가 라이브 사이트를 안 망가뜨리는가"에 달렸다. 사람 검수 게이트 또는
   수정가능구역 제약 없이 auto-deploy하면 한 번의 사고로 신뢰 붕괴. (창업자 동의)

## Approaches Considered
### Approach A: 사람이 게이트, AI가 손발 (human-in-the-loop)
완전자동 배포 없음. 요청 → AI가 변경안+미리보기 자동 생성 → 창업자 원클릭 승인 → 배포.
노동 90%를 AI가, 사람은 판단만. 노력 S / 위험 낮음. 이번 주에 과금 시작 가능.

### Approach B: 셀프서비스 + 가드레일
고객이 직접 포털에서 입력 → 비주얼 before/after 미리보기 → 승인 → 자동배포+자동롤백+이력.
핵심 안전장치 = 개발자가 지정한 "수정 가능 구역(텍스트·사진·영업시간·가격)" 밖은 AI가 못 건드림.
노력 L / 위험 중. 확장 가능한 SaaS.

### Approach C: 에디터 대신 카톡 봇
새 UI 학습 0. 고객이 이미 쓰는 카톡 채널에 "영업시간 10시로 바꿔줘" → AI 변경안 제시 →
톡 버튼 확인 → 배포. 유통 = 이미 쓰는 채널. 노력 M / 위험 중.

## Recommended Approach
**A + C 하이브리드를 MVP로.**
- 인터페이스: C(카톡) — 단, **초기엔 기존 1:1 카톡 + 창업자 수동 중계**로 시작. 카카오 비즈니스
  채널/챗봇/알림톡은 사업자인증·채널심사·템플릿 사전승인이 필요해 1주 안에 양방향 봇이 안 뜸.
  자동 봇은 후행. 고객 경험상으론 똑같이 "카톡으로 보내면 됨".
- 백엔드: A(사람 게이트) — AI가 변경안 생성 + **before/after 스크린샷 1장**으로 미리보기,
  창업자가 폰에서 승인. **배포 = 창업자가 직접 푸시**(MVP는 CI 자동화 안 함).
- 미리보기 형식: 비전문가는 코드 diff 못 읽으므로 **항상 렌더된 before/after 스크린샷**으로 고정.
- 롤백: **git 1커밋 = 1수정** → 사고 시 `git revert`로 즉시 원복. (Constraints의 "롤백 불가"를
  이 한 줄로 해소.)
- 진화 경로: 고객 늘면 B의 "수정 가능 구역" 가드레일을 도입해 점진적 자동화(저위험 수정부터
  사람 게이트 제거).
- 근거: 신뢰 리스크 0으로 시작, 과금 검증 최속, 빌드 최소. 진엽 형 GraBis의 "0원 집계(반복은
  코드, 판단은 사람)" + "escape hatch" 패턴을 웹 유지보수에 그대로 이식.

## Open Questions
- 가격: 월 구독 얼마? (수정 빈도·사이트 규모 기준 단일가 vs 티어) — 검증으로 확정.
- 한 사이트당 월 수정 요청 실제 분포는? (창업자가 데이터를 이미 가짐 — 카톡 로그)
- 카톡 채널 봇 vs 알림톡 vs 단순 웹폼 — 초기엔 가장 적은 빌드로.
- 사이트 호스팅·배포 파이프라인 표준화(고객마다 다르면 자동화 불가) → 단일 템플릿/스택 강제 필요.
- AI 오작동 시 책임·롤백 SLA를 구독 약관에 어떻게 명시?

## Success Criteria
- **1주 내**: 기존 2고객 중 최소 1명이 월 구독료를 *실제로 결제*(정기이체 or 토스 정기결제).
  (핵심 demand gate. 빌드 0, 검증만.)
- **1개월 내 (게이트 분리)**:
  - (a) 카톡 요청 → 창업자 수동 처리 → 배포 루프로 실제 수정 5건 이상 처리(AI 없이도 OK).
  - (b) 그중 최소 1~2건을 AI 변경안 생성으로 시도해 파이프라인 난이도 실측.
  → AI 자동화는 "되면 좋은 것"이지 1개월 게이트의 필수 아님. 매출/의존이 먼저.
- **분기 내**: 소개로 유료 고객 3+ 확장. 창업자의 수정 1건당 손시간 80%↓(AI 자동화 본격 도입).

## Distribution Plan
- 초기 유통 = 창업자의 기존 관계 + 소개(미용실 원장 → 동종 자영업, 진엽 형 → 네트워크).
- 제품 전달 = 웹 서비스(고객은 카톡으로 접근, 백오피스는 창업자 대시보드). 신규 바이너리/앱 없음.
- 배포 파이프라인: 고객 사이트 스택을 단일 템플릿으로 표준화해야 AI 자동수정+CI 재배포가 성립.
  (표준화 전까지는 A의 사람 게이트가 이 빈틈을 메움.)

## Dependencies
- 고객 사이트 스택 표준화(템플릿/프레임워크 고정)가 자동화의 선행조건.
- 카톡 채널/봇 또는 알림톡 채널 셋업.
- AI 변경 생성 + 비주얼 미리보기 렌더 파이프라인.

## The Assignment
**이번 주 안에, 미용실 원장님과 진엽 형 회사에 가격표를 들이밀어라.**
"앞으로 홈페이지 수정은 월 N만원에 무제한으로 관리해드릴게요." 이 한 문장이 세상에서 제일 싸고
빠른 수요 테스트다. 코드 한 줄 더 짜기 전에 이걸 먼저 하라. 둘 다 YES면 제품을 지을 자격이 생긴
거고, 친구인데도 NO면 모르는 고객은 100% NO다. 결과를 가지고 다시 오면 다음 단계를 짠다.

함께 할 것(빌드 0, 다 조사·확인):
- **결제 수단 1개 정하기**: 1주 안에 받으려면 수동 정기이체 or 토스 정기결제. 1회 입금과
  구독을 구분할 수 있어야 함.
- **기존 2사이트의 스택 확인** (가장 중요): 무슨 프레임워크/호스팅으로 만들었나? 표준 스택이
  아니면 나중에 AI 자동수정이 첫 고객부터 안 된다. (리뷰어 최우선 지적)
- **지난 3개월 카톡 수정 요청 개수 세기**: 그 빈도가 곧 가격 근거이자 "무제한"의 실제 비용.

## What I noticed about how you think
- 너는 "소상공인"이라고 안 했어. "내가 다니는 미용실 원장님", "진엽이형네 회사"라고 했지.
  카테고리가 아니라 이름. 그 구체성은 드물어.
- 내가 깐 전제 ②에 그냥 안 끄덕이고 "지인은 진짜 고객이 아니다"라고 받아쳤어. 그 반골이
  창업자의 자질이야. (절반만 맞았지만, 틀린 절반을 스스로 고칠 수 있는 사람이었어.)
- 너는 이미 만들어서 납품했어. 기획만 하는 사람이 아니라 출하하는 사람이야. 그게 제일 귀해.
